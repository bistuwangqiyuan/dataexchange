# Feature Specification: 加密货币在线交易平台 (Crypto Trading Platform)

**Feature Branch**: `001-description-netlify-bianca`  
**Created**: 2025-10-11  
**Status**: Draft  
**Input**: User description: "开发比特币在线交易平台系统，该系统为网站版本，部署到netlify，具有丰富、合理的内容，网站的目标是类似Bianca的加密货币在线交易平台。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 用户注册与身份验证 (Priority: P1)

新用户访问平台，需要创建账户以开始交易加密货币。用户通过提供电子邮件、设置密码和基本个人信息来注册账户，完成邮箱验证后即可登录系统。

**Why this priority**: 这是平台的基础功能，没有用户账户系统，所有其他功能都无法使用。这是最小可行产品（MVP）的核心。

**Independent Test**: 可以通过创建新用户账户、接收验证邮件、登录系统来独立测试。即使没有交易功能，用户账户系统本身就能提供价值（为后续功能奠定基础）。

**Acceptance Scenarios**:

1. **Given** 用户在注册页面, **When** 用户输入有效的邮箱、密码（至少8位包含字母和数字）和基本信息并提交, **Then** 系统创建账户并发送验证邮件
2. **Given** 用户收到验证邮件, **When** 用户点击验证链接, **Then** 账户被激活，用户可以登录
3. **Given** 已注册用户在登录页面, **When** 用户输入正确的邮箱和密码, **Then** 用户成功登录并进入个人仪表板
4. **Given** 用户忘记密码, **When** 用户使用"忘记密码"功能并提供注册邮箱, **Then** 用户收到密码重置邮件

---

### User Story 2 - 实时市场行情查看 (Priority: P1)

用户登录后可以查看主要加密货币的实时价格、24小时价格变化、交易量和价格走势图表（K线图）。用户可以选择不同的时间周期（1分钟、5分钟、1小时、1天）查看历史价格走势。

**Why this priority**: 查看市场行情是交易决策的基础，是用户访问交易平台的核心需求之一。这个功能可以独立提供价值，即使用户还不能交易，也能获取市场信息。

**Independent Test**: 可以通过访问行情页面、查看不同加密货币的价格和图表、切换时间周期来独立测试。无需交易功能即可验证。

**Acceptance Scenarios**:

1. **Given** 用户在市场页面, **When** 页面加载, **Then** 显示BTC、ETH、USDT等主要加密货币的当前价格和24小时涨跌幅
2. **Given** 用户选择某个加密货币, **When** 点击查看详情, **Then** 显示该货币的详细K线图和交易深度信息
3. **Given** 用户查看K线图, **When** 切换时间周期（1分钟/1小时/1天）, **Then** 图表更新显示对应周期的历史数据
4. **Given** 市场价格发生变化, **When** 价格更新, **Then** 页面实时刷新显示最新价格（无需手动刷新）

---

### User Story 3 - 数字钱包管理 (Priority: P2)

用户拥有多币种数字钱包，可以查看各种加密货币的余额、充值地址。用户可以通过充值功能将加密货币转入平台钱包，也可以通过提现功能将加密货币转出到外部钱包地址。

**Why this priority**: 钱包功能是交易的前提条件，用户需要先充值才能开始交易。但在MVP阶段，可以先提供模拟余额让用户体验交易流程。

**Independent Test**: 可以通过查看钱包余额、生成充值地址、提交提现申请来独立测试。可以使用测试网络或模拟数据验证流程。

**Acceptance Scenarios**:

1. **Given** 用户在钱包页面, **When** 页面加载, **Then** 显示所有支持的加密货币及其当前余额（包括余额为0的货币）
2. **Given** 用户选择某个加密货币充值, **When** 点击充值按钮, **Then** 系统生成该货币的充值地址和二维码
3. **Given** 用户选择某个加密货币提现, **When** 用户输入提现地址、金额并确认, **Then** 系统验证地址格式，创建提现申请并要求二次验证（邮件或2FA）
4. **Given** 提现申请已提交, **When** 用户确认二次验证, **Then** 提现申请进入处理队列，用户可以在历史记录中查看状态

---

### User Story 4 - 现货交易功能 (Priority: P1)

用户可以进行加密货币现货交易，包括买入和卖出操作。支持市价单（按当前市场价格立即成交）和限价单（设置期望价格，等待市场价格达到时自动成交）两种交易类型。

**Why this priority**: 交易功能是交易平台的核心价值所在，这是用户使用平台的主要目的。虽然标记为P1，但在实现顺序上应在用户系统和行情系统之后。

**Independent Test**: 可以通过创建买单/卖单、查看订单状态、取消订单来独立测试。可以使用模拟资金进行测试交易。

**Acceptance Scenarios**:

1. **Given** 用户在交易页面且有足够余额, **When** 用户选择交易对（如BTC/USDT）、选择"买入"、选择"市价单"并输入数量, **Then** 订单立即按当前市场价格成交，余额相应变化
2. **Given** 用户在交易页面, **When** 用户选择"限价单"、输入期望价格和数量并提交, **Then** 订单进入挂单队列，显示在"当前委托"列表中
3. **Given** 用户有未成交的限价单, **When** 市场价格达到限价单价格, **Then** 订单自动成交，用户收到成交通知
4. **Given** 用户有未成交的限价单, **When** 用户点击"取消订单", **Then** 订单被取消，冻结的资金解冻并返回可用余额

---

### User Story 5 - 订单与交易历史管理 (Priority: P2)

用户可以查看所有交易订单的历史记录，包括当前未成交的委托订单和已完成的历史订单。每条记录显示交易对、交易类型、价格、数量、时间和状态。

**Why this priority**: 订单管理帮助用户追踪交易活动和资金变动，对于交易决策和账户管理很重要，但不是最初MVP的必需功能。

**Independent Test**: 可以通过创建多个订单后查看订单列表、筛选不同状态的订单、查看订单详情来独立测试。

**Acceptance Scenarios**:

1. **Given** 用户在订单页面, **When** 选择"当前委托"标签, **Then** 显示所有未成交的限价单，包括交易对、价格、数量、已成交数量、状态
2. **Given** 用户在订单页面, **When** 选择"历史订单"标签, **Then** 显示所有已完成（完全成交或已取消）的订单
3. **Given** 用户查看历史订单, **When** 使用筛选器（按时间范围、交易对、状态）, **Then** 订单列表根据筛选条件更新
4. **Given** 用户点击某个订单, **When** 查看订单详情, **Then** 显示完整的订单信息，包括成交明细（部分成交的情况）

---

### User Story 6 - 账户安全与设置 (Priority: P2)

用户可以配置账户安全设置，包括启用两因素认证（2FA）、设置交易密码、查看登录历史。用户可以管理个人信息和偏好设置（如语言、时区、通知偏好）。

**Why this priority**: 安全功能对于金融平台至关重要，但可以在基本交易功能实现后再添加。初期可以先提供基本的密码登录。

**Independent Test**: 可以通过启用/禁用2FA、修改交易密码、查看登录记录来独立测试。

**Acceptance Scenarios**:

1. **Given** 用户在安全设置页面, **When** 用户启用2FA功能, **Then** 系统生成2FA密钥和二维码，用户扫描后输入验证码完成绑定
2. **Given** 用户已启用2FA, **When** 用户登录或进行敏感操作（如提现）, **Then** 系统要求输入2FA验证码
3. **Given** 用户在安全设置页面, **When** 用户设置交易密码（6位数字）, **Then** 后续交易时需要输入交易密码才能提交订单
4. **Given** 用户在账户设置页面, **When** 查看登录历史, **Then** 显示最近的登录记录，包括时间、IP地址、设备信息

---

### User Story 7 - 资产概览仪表板 (Priority: P3)

用户登录后的首页显示资产概览仪表板，包括总资产价值（折合为USDT或USD）、各币种资产分布饼图、近期盈亏统计、待处理的订单数量、市场热门币种等信息。

**Why this priority**: 仪表板提供整体资产视图，提升用户体验，但不是核心交易功能，可以在后期优化阶段添加。

**Independent Test**: 可以通过登录后查看仪表板、验证各项数据准确性来独立测试。

**Acceptance Scenarios**:

1. **Given** 用户登录后, **When** 进入仪表板页面, **Then** 显示总资产价值和24小时盈亏百分比
2. **Given** 用户在仪表板, **When** 查看资产分布图, **Then** 显示各加密货币的持仓比例饼图或柱状图
3. **Given** 用户在仪表板, **When** 查看"待处理订单"区域, **Then** 显示当前未成交的订单数量和快捷链接
4. **Given** 用户在仪表板, **When** 查看"市场热点"区域, **Then** 显示24小时涨幅最高的加密货币列表

---

### Edge Cases

- **网络连接中断**: 当用户在交易过程中网络中断，系统如何处理？已提交的订单状态如何确认？
- **余额不足**: 当用户尝试下单但余额不足（考虑手续费后）时，系统如何提示？
- **价格剧烈波动**: 当市场价格在订单提交和执行之间发生剧烈变化，如何保护用户利益？
- **并发交易**: 当用户在多个设备或浏览器标签页同时操作，如何防止余额超支或重复交易？
- **无效地址**: 当用户输入错误的提现地址格式时，系统如何验证和阻止？
- **大额交易**: 是否需要对单笔交易或单日交易量设置上限？
- **账户冻结**: 当检测到可疑活动时，如何冻结账户并通知用户？
- **市场深度不足**: 当用户的市价单数量超过当前市场可成交量时，如何处理？
- **精度问题**: 加密货币有不同的小数位精度（如BTC 8位，ETH 18位），如何正确处理计算和显示？
- **时区差异**: 不同地区用户如何正确显示交易时间？

## Requirements *(mandatory)*

### Functional Requirements

**用户管理模块**:

- **FR-001**: 系统必须允许新用户通过邮箱和密码注册账户
- **FR-002**: 系统必须验证邮箱地址的有效性并发送验证邮件
- **FR-003**: 系统必须验证密码强度（至少8位，包含字母和数字）
- **FR-004**: 系统必须提供密码重置功能（通过邮箱验证）
- **FR-005**: 系统必须提供用户登录和登出功能
- **FR-006**: 系统必须支持会话管理，保持用户登录状态
- **FR-007**: 系统必须提供两因素认证（2FA）功能（可选启用）
- **FR-008**: 系统必须记录用户的登录历史（时间、IP、设备）

**市场行情模块**:

- **FR-009**: 系统必须显示主要加密货币的实时价格（BTC、ETH、USDT、BNB、ADA、XRP等）
- **FR-010**: 系统必须显示每个加密货币的24小时价格变化百分比和交易量
- **FR-011**: 系统必须提供K线图展示，支持多种时间周期（1m、5m、15m、1h、4h、1d、1w）
- **FR-012**: 系统必须[NEEDS CLARIFICATION: 实时行情数据源 - 使用真实交易所API（如Binance API）还是模拟数据？这将影响系统架构、成本和数据准确性]
- **FR-013**: 系统必须以合理的频率更新行情数据（建议：价格每3-5秒更新一次，K线图按周期更新）
- **FR-014**: 系统必须提供交易深度图（买卖盘口数据）

**钱包管理模块**:

- **FR-015**: 系统必须为每个用户维护多币种钱包（至少支持BTC、ETH、USDT）
- **FR-016**: 系统必须显示每种加密货币的可用余额和冻结余额（委托订单占用）
- **FR-017**: 系统必须[NEEDS CLARIFICATION: 充值/提现功能实现方式 - 连接真实区块链网络（需要钱包服务和节点）、使用第三方托管API、还是仅提供模拟充值功能？这影响技术复杂度、安全性和合规要求]
- **FR-018**: 系统必须生成唯一的充值地址（或提供充值说明）
- **FR-019**: 系统必须验证提现地址格式的有效性
- **FR-020**: 系统必须对提现操作要求二次验证（邮件验证码或2FA）
- **FR-021**: 系统必须记录所有充值和提现历史

**交易模块**:

- **FR-022**: 系统必须支持现货交易（买入和卖出）
- **FR-023**: 系统必须支持市价单（按当前最优价格立即成交）
- **FR-024**: 系统必须支持限价单（用户设定价格，等待市场价格达到时成交）
- **FR-025**: 系统必须在下单前验证用户余额是否足够（包括交易手续费）
- **FR-026**: 系统必须在下单成功后冻结相应的资金（限价单）或立即扣除（市价单）
- **FR-027**: 系统必须计算并收取交易手续费（建议：0.1%-0.2%）
- **FR-028**: 系统必须提供订单取消功能（仅限未成交或部分成交的限价单）
- **FR-029**: 系统必须在订单成交后立即更新用户余额
- **FR-030**: 系统必须[NEEDS CLARIFICATION: 订单撮合机制 - 使用真实的订单撮合引擎（需要订单簿和匹配算法）还是直接对接外部交易所API？这影响系统的复杂度和是否需要管理流动性]

**订单管理模块**:

- **FR-031**: 系统必须显示用户的当前委托订单（未完全成交的订单）
- **FR-032**: 系统必须显示用户的历史订单（已完成或已取消的订单）
- **FR-033**: 系统必须提供订单筛选功能（按交易对、状态、时间范围）
- **FR-034**: 系统必须显示每个订单的详细信息（交易对、类型、价格、数量、已成交数量、状态、时间）
- **FR-035**: 系统必须在订单状态变化时通知用户（可通过站内通知或邮件）

**安全模块**:

- **FR-036**: 系统必须对所有密码进行加密存储（使用bcrypt或类似安全哈希算法）
- **FR-037**: 系统必须实施防暴力破解机制（登录失败次数限制）
- **FR-038**: 系统必须提供交易密码功能（与登录密码分离）
- **FR-039**: 系统必须对敏感操作（提现、修改密码、禁用2FA）进行二次验证
- **FR-040**: 系统必须在检测到异常活动时发送警报邮件
- **FR-041**: 系统必须实施API请求频率限制，防止滥用

**用户界面模块**:

- **FR-042**: 系统必须提供响应式设计，支持桌面和移动设备
- **FR-043**: 系统必须支持至少两种语言（中文和英文）
- **FR-044**: 系统必须提供深色和浅色主题切换
- **FR-045**: 系统必须在所有表单中提供清晰的错误提示和验证信息

### Key Entities

- **User（用户）**: 代表平台用户账户，包含邮箱、密码（加密）、注册时间、KYC状态、2FA配置、交易密码、语言偏好等属性
  
- **Wallet（钱包）**: 代表用户的加密货币钱包，每个用户有多个钱包（每种币一个），包含币种类型、可用余额、冻结余额、充值地址等属性

- **Order（订单）**: 代表一笔交易订单，包含用户ID、交易对（如BTC/USDT）、订单类型（市价/限价）、交易方向（买入/卖出）、价格、数量、已成交数量、订单状态（待成交/部分成交/完全成交/已取消）、创建时间、更新时间等属性

- **Transaction（交易记录）**: 代表订单的实际成交记录，一个订单可能产生多条交易记录（部分成交情况），包含订单ID、成交价格、成交数量、手续费、成交时间等属性

- **WalletHistory（钱包历史）**: 代表钱包的所有资金变动记录，包含用户ID、币种、变动类型（充值/提现/交易买入/交易卖出/手续费）、变动金额、余额快照、时间、关联订单ID等属性

- **MarketPrice（市场价格）**: 代表某个交易对的当前市场价格和统计信息，包含交易对、当前价格、24小时最高价、24小时最低价、24小时涨跌幅、24小时交易量、更新时间等属性

- **SecurityLog（安全日志）**: 代表用户的安全相关操作记录，包含用户ID、操作类型（登录/登出/修改密码/提现等）、IP地址、设备信息、操作结果、时间等属性

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 新用户能够在3分钟内完成注册流程（从进入注册页面到收到验证邮件）
- **SC-002**: 用户能够在2秒内看到最新的市场价格更新（行情页面加载时间）
- **SC-003**: 市价单能够在1秒内完成撮合和余额更新
- **SC-004**: 限价单能够在价格到达后的5秒内自动成交
- **SC-005**: 系统能够支持至少100个并发用户同时交易而不出现延迟或错误
- **SC-006**: 用户能够在任何设备（桌面、平板、手机）上流畅使用所有核心功能，页面加载时间不超过3秒
- **SC-007**: 95%的交易操作能够在用户首次尝试时成功完成（无需重试）
- **SC-008**: 用户账户安全功能（2FA、交易密码）的启用率达到60%以上
- **SC-009**: 系统正常运行时间达到99.5%（即每月停机时间不超过3.6小时）
- **SC-010**: 所有密码相关功能（注册、登录、重置）必须在5次失败尝试后触发保护机制
- **SC-011**: 用户能够在钱包页面看到精确到对应币种标准精度的余额（BTC 8位小数，ETH 18位小数）
- **SC-012**: 交易历史和订单记录能够在3秒内检索并展示最近6个月的数据
- **SC-013**: 新功能上线后，用户支持请求（关于该功能的问题）在第一周内不超过用户总数的5%
- **SC-014**: 用户能够通过少于3次点击完成任何核心操作（查看行情、下单、查看余额）

## Assumptions

以下是基于用户需求和行业标准做出的合理假设：

1. **部署环境**: 项目将部署到Netlify，采用Jamstack架构，前端为静态站点，后端API通过Netlify Functions或Supabase Functions实现

2. **技术栈**: 使用现代Web技术栈（如Astro/Next.js + React + Tailwind CSS），Supabase作为数据库和认证服务

3. **数据持久化**: 使用Supabase PostgreSQL存储用户数据、订单、交易历史等

4. **支持的加密货币**: 初期支持主流加密货币（BTC、ETH、USDT、BNB、ADA、XRP），后续可扩展

5. **交易手续费**: 采用行业标准手续费率（买入和卖出各0.1%-0.2%）

6. **认证方式**: 主要使用邮箱+密码认证，2FA为可选增强安全功能

7. **语言支持**: 初期支持中文和英文，界面设计考虑多语言扩展性

8. **时区处理**: 所有时间戳使用UTC存储，前端根据用户本地时区显示

9. **数据精度**: 价格和数量计算使用高精度数学库（如decimal.js），避免浮点数精度问题

10. **会话管理**: 用户会话有效期为7天（可配置），支持"记住我"功能

11. **性能目标**: 针对中小规模用户（100-1000并发），不是高频交易系统

12. **合规性**: 假设这是教育或演示项目，不涉及真实金融监管要求。如果需要处理真实资金，需要额外的KYC/AML合规流程

13. **测试策略**: 遵循TDD原则，测试覆盖率目标≥80%，包括单元测试、集成测试和E2E测试

14. **浏览器支持**: 支持现代浏览器的最新两个版本（Chrome、Firefox、Safari、Edge）

## Out of Scope

以下功能明确不在本次开发范围内：

1. **高级交易功能**: 杠杆交易、期货合约、期权交易、永续合约
2. **法币支付**: 信用卡充值、银行转账、PayPal等法币充值渠道
3. **完整KYC流程**: 身份证验证、人脸识别、地址证明等正式金融机构要求的KYC流程
4. **社交功能**: 用户之间的聊天、跟单、社区讨论功能
5. **移动原生应用**: iOS/Android原生应用（仅提供响应式网页版）
6. **API交易接口**: 供第三方程序接入的REST API或WebSocket API
7. **量化交易工具**: 策略回测、程序化交易、API密钥管理
8. **多账户管理**: 子账户、团队账户、企业账户
9. **积分奖励系统**: 推荐奖励、交易返佣、VIP等级系统
10. **法律合规工具**: 税务报告、审计日志导出、监管报告生成
11. **客服系统**: 在线客服聊天、工单系统
12. **高级安全功能**: 防钓鱼码、白名单地址、提现时间锁

## Dependencies

1. **外部服务**:
   - Supabase（数据库、认证、实时功能）
   - Netlify（托管和无服务器函数）
   - 市场数据API（待明确，可能是Binance API、CoinGecko API或自建模拟数据）
   - 邮件服务（Supabase内置或SendGrid/AWS SES）

2. **技术依赖**:
   - 前端框架（建议Astro或Next.js）
   - UI组件库（建议Tailwind CSS + Headless UI或shadcn/ui）
   - 图表库（TradingView Lightweight Charts或ECharts）
   - 加密库（bcrypt.js）
   - 高精度数学库（decimal.js或big.js）

3. **开发工具**:
   - 测试框架（Vitest、Jest或Playwright）
   - 代码质量工具（ESLint、Prettier、TypeScript）
   - 版本控制（Git）

## Risks & Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| 实时数据源不稳定或API限流 | 高 - 影响核心功能 | 中 | 实施缓存策略、使用多个数据源备份、实施降级方案 |
| 交易过程中的数据一致性问题 | 高 - 可能导致资金错误 | 中 | 使用数据库事务、实施严格的余额验证、添加详细日志 |
| 安全漏洞（SQL注入、XSS等） | 高 - 可能导致用户资产损失 | 中 | 代码审查、使用参数化查询、输入验证、定期安全扫描 |
| 性能瓶颈（高并发场景） | 中 - 影响用户体验 | 中 | 负载测试、数据库索引优化、实施缓存、CDN加速 |
| 用户资金丢失或错误扣除 | 高 - 信任和法律问题 | 低 | 详细的交易日志、余额快照、事务回滚机制、充分测试 |
| 市场价格剧烈波动导致交易争议 | 中 - 用户投诉 | 中 | 清晰的交易确认流程、滑点保护机制、交易记录可追溯 |
| Netlify部署限制（函数执行时间、调用次数） | 中 - 可能需要调整架构 | 中 | 评估Netlify限制、设计高效的函数、考虑备用方案 |
| 浮点数精度问题导致计算错误 | 高 - 影响资金准确性 | 高 | 使用高精度数学库、所有金额使用字符串或整数存储 |
| 跨浏览器兼容性问题 | 低 - 部分用户无法访问 | 低 | 使用现代框架、Polyfills、浏览器测试 |
| 第三方服务中断（Supabase、Netlify） | 中 - 服务不可用 | 低 | 监控告警、备份策略、多区域部署（如可能） |
